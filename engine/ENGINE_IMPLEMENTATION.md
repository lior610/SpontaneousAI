# Attraction Engine Implementation Details

This document outlines the implementation of the three main layers of the attraction engine: Cluster Retrieval, Ranking, and Real-Time Feedback.

## 1. Cluster-Diverse Retrieval
**Core Files:**
- `engine/src/services/cluster_retrieval.py`
- `engine/src/db/cluster_queries.py`

Instead of picking the top 10 most similar attractions overall (which often results in a list of 10 coffee shops in the same neighborhood), the engine utilizes the `location_cluster_id` stored alongside pgvector's embeddings to guarantee geographic and categorical diversity.

### How it Works:
1. **Geographic and Temporal Pre-Filtering (`cluster_queries.py:execute_cluster_similarity_query`)**: Before running heavy vector math, the database instantly calculates a 1.0x geographic Bounding Box from the user's GPS coordinates (`max_walk_km`). Next, it uses PostgreSQL's native `SPLIT_PART` math to parse the `"HH:MM-HH:MM"` hours string. If an attraction is mathematically outside the bounding box or verified as closed at the `current_hour`, it is immediately dropped to preserve database search capacity.
2. **Querying the Vector Space (`cluster_queries.py`)**: The engine runs a single optimized PostgreSQL query against the entire vector space for the user's location (e.g., Paris), evaluating only the open and local candidates.
3. **Window Functions (`cluster_queries.py`)**: We use PostgreSQL's `ROW_NUMBER() OVER(PARTITION BY location_cluster_id ORDER BY embedding <=> user_vector)` to group the most semantically similar attractions by their geographic cluster.
4. **Filtering (`cluster_retrieval.py:get_candidate_pool`)**: The query limits results to the top `M` (e.g., 5) attractions per cluster, and then we truncate the total clusters to the top `N` (e.g., 5). This yields a diverse Candidate Pool of ~25 attractions from different parts of the city.
5. **Exclusions (`engine/src/db/feedback_queries.py:get_excluded_place_ids`)**: Any attraction the user has already `liked`, `skipped`, or `visited` during this trip is explicitly excluded at the SQL level via a `NOT IN` clause.

## 2. Ranking Layer
**Core Files:**
- `engine/src/services/ranking_service.py`

The Candidate Pool generated by the retrieval layer is then passed to the `RankingEngine` to determine the absolute logical ordering for the user right now.

### Scoring Formula (`ranking_service.py:rank_candidates`):
The final score is a pure mathematical calculation (no runtime ML inference) weighted according to the system configuration:
- **`semantic_score` (40%)**: The cosine similarity (0.0 to 1.0) directly from the vector search.
- **`distance_score` (15%)**: Computed via the Haversine formula from the user's current GPS location. Scaled as `max(0, 1 - (dist_km / max_walk_km))`.
- **`popularity_score` (20%)**: Sourced directly from Foursquare APIs. Scales naturally from 0.0 to 1.0 based on user reviews and traction metrics. It defaults to 0.2 if the place is completely unknown.
- **`hours_score` (10%)**: A fallback binary multiplier. Because explicit opening hours are evaluated instantly in the SQL layer via `SPLIT_PART`, any attractions that make it up to the Python `RankingEngine` are likely open. However, if an attraction slipped through with a messy string format (like `12 PM`), Python re-evaluates it here. If verifiably closed, the entire candidate's `final_score` drops to absolutely `0.0`.
- **`budget_score` (10%)**: `1.0` if the attraction budget matches the user's `travel_style`, otherwise scaled down.
- **`diversity_bonus` (5%)**: This score is applied dynamically during ranking to promote variety. If an attraction introduces a new category not already in the top recommended results, it gets a slight bump. Additionally, it acts as a **cluster counter penalty**: if we've already recommended many attractions from the exact same `location_cluster_id`, subsequent attractions from that cluster receive a negative penalty, forcing the top results to organically spread across different neighborhoods.

## 3. Real-Time Feedback Loop
**Core Files:**
- `engine/src/services/feedback_service.py`
- `engine/src/services/preference_service.py`
- `engine/src/db/preference_queries.py`

The engine dynamically adjusts recommendations mid-trip without requiring heavy backend tasks or ML training.

### The 5-File Architecture Pipeline:
1. **The API Entry Point (`internal-routes/recommendations.py`)**: The `POST /recommendations/feedback` endpoint receives the user's action (`"liked"`, `"skipped"`, or `"visited"`) along with the specific `place_id`.
2. **The Business Logic Manager (`services/feedback_service.py`)**: The `process_feedback()` function decides what to do next. It logs the action to the DB immediately, and if (and only if) the action was `"liked"`, it triggers the mathematical AI Vector update.
3. **The Database Logger (`db/feedback_queries.py`)**: This file handles the raw SQL connection. It contains the `INSERT` query that permanently stamps the action into the `trip_feedback` table. It also contains `get_excluded_place_ids`, which blocks them from ever being recommended again.
4. **The EMA Math Engine (`services/preference_service.py`)**: Inside `PreferenceComposer.apply_feedback()`, the engine calculates the **Exponential Moving Average (EMA)**. It dynamically pulls the user's core trip vector mathematically closer to the vector of the attraction they just liked in 384-dimensional space.
5. **The State Saver (`db/preference_queries.py`)**: `upsert_preference_embedding` physically saves the new, dynamically evolved vector back into the `user_preference_embeddings` PostgreSQL table. This fundamentally overwrites their old profile, completely changing the results of their next API call!

## 5. Q&A

### Cluster-Diverse Retrieval

**Q: Can you explain the vector query step by step?**
```sql
(1 - (embedding <=> %s::vector) / 2) as similarity,
(embedding <=> %s::vector) as distance,
ROW_NUMBER() OVER(
    PARTITION BY location_cluster_id 
    ORDER BY embedding <=> %s::vector
) as cluster_rank
FROM attractions
```
- **`embedding <=> %s::vector`**: The `<=>` operator is `pgvector`'s built-in function for calculating Cosine Distance between the attraction's embedding and the user's preference vector. A distance of `0` means they are perfectly identical, and `2` means they are complete opposites.
- **`similarity`**: We convert the Cosine Distance into a Cosine Similarity percentage (0 to 1). The formula `(1 - distance / 2)` normalizes it so `1.0` means a perfect match.
- **`ROW_NUMBER() OVER(PARTITION BY ...)`**: This is the magic "clustering" step. PostgreSQL groups all the rows by their `location_cluster_id`. Inside each group (partition), it sorts them by how close they match the user (`ORDER BY embedding <=> user_vector`). It then assigns a sequential rank (1, 2, 3...) to each attraction in that specific cluster.

**Q: What are the `params` in the SQL execution?**
```python
params: List[Any] = [embedding_str, embedding_str, embedding_str, location_id]
```
These parameters map directly to the `%s` placeholders in the raw SQL query above. Because we reference the user's embedding vector three separate times in the internal `SELECT` statement (once for similarity, once for distance, and once in the `ORDER BY` clause), we must pass the exact same string into the `execute` parameters array three times. The fourth parameter is the `location_id` for the `WHERE` clause.

**Q: Is the hard-filtering part necessary if we take care of it in the formula later?**
Technically, if we want the engine to simply drop the score to `0` for closed attractions, we **should not** filter them out during the SQL query. We only want to use these SQL `filters` for strict, non-negotiable constraints (e.g., minimum age restrictions, wheelchair accessibility). By default, we are passing `filters=None` from the router right now, so it has no effect, but it's good to keep this hook available for strict constraints later!

**Q: Can you explain FinalRanking? How many attractions you get from each cluster? how you get the min_dist value?**
```sql
FinalRanking AS (
    SELECT r.*,
           DENSE_RANK() OVER(ORDER BY c.min_dist) as cluster_score_rank
    FROM RankedAttractions r
    JOIN ClusterMins c ON r.location_cluster_id = c.location_cluster_id
)
```
This solves the problem of "Which clusters are the best overall?". 
- Earlier, we figured out the #1 best attraction for *every single cluster in the city* (which is the row where `cluster_rank = 1`). We stored that single minimum distance value in the `ClusterMins` CTE.
- Here, we `JOIN` the attractions back against those minimums.
- `DENSE_RANK() OVER(ORDER BY c.min_dist)` ranks the *clusters themselves* based on that minimum distance. The cluster containing the single best matching attraction in the entire city becomes Cluster Rank 1. The cluster with the second-best match becomes Cluster Rank 2, and so on. 
- **Number of Attractions Retreived**: The query returns a max of `top_per_cluster` attractions (e.g., 5) from each of the Top `max_clusters` (e.g., 5) clusters. This yields up to 25 candidates total natively from the database query.

**Q: What are the functions imported mid-file in cluster_retrieval.py and why isn't the import at the beginning?**
```python
try:
    from db.usersConnection import get_db_connection as get_users_conn
```
This is a standard backend pattern to avoid **Circular Imports**. Because `cluster_retrieval.py` is in the `attractions` context but needs to briefly peek into the `client_info` database to see what places the user has already visited, importing it at the very top of the file can sometimes crash FastAPI if the two modules try to import each other simultaneously during statup. Putting the import *inside* the function ensures it only loads right when the API is actually called.

### Ranking Layer

**Q: Explain why Haversine is needed?**
The database stores standard GPS coordinates (Latitude and Longitude) representing points on a sphere (the Earth). You cannot use standard straight-line math (Pythagorean theorem) to calculate the distance between two GPS coordinates because the earth is curved! The **Haversine Formula** uses spherical trigonometry to calculate the exact real-world walking distance in kilometers between the user and the attraction.

**Q: All the distances are 0 in the results. Why?**
In the initial test, the `current_location` was passed as `{lat: 0, lng: 0}` or was mapped to the exact same default point. When we explicitly sent the London coordinates inside the request payload (`"current_location": {"lat": 51.5072, "lng": -0.1276}`), the Haversine formula correctly calculated the exact real-world distances between that point and the London attractions (e.g., `2.8 km`, `13.2 km`), so the algorithm works perfectly based directly on the GPS payload!

**Q: How is geographic distance filtered efficiently?**
Because checking exact Haversine distances against millions of AI vectors would completely crash the servers, distance filtering uses a **two-layer funnel**:
1. **Stage 1: SQL Bounding Box (`cluster_queries.py`)**: The database translates the `max_walk_km` radius (multiplied by a `1.0` soft-boundary leniency) into pure Latitude/Longitude degree offsets. It draws a massive literal square (not a circle) around the user and deletes anything outside the `BETWEEN` parameter. Because corners of squares are longer than sides, locations mathematically on the diagonal boundary (up to ~1.4x distance) uniquely survive this pre-filter.
2. **Stage 2: Python Haversine ("The Guillotine")**: Surviving candidates enter Python, which executes the mathematically perfect Earth-curved *Haversine Formula*. Python then applies a rigid floor constraint: `max(0.0, 1.0 - (dist / max_walk_km))`. If an attraction is mathematically even 1 meter further than the exact limit (e.g. `2.01km`), its distance score hits `0.0`. This forces distant attractions to rely purely on their AI Semantic vector strength to survive the rankings.

**Q: How does Popularity influence the recommendations?**
We integrated a `popularity` metric sourced directly from the Foursquare API data (scaled from 0.0 to 1.0). In the Ranking Formula, popularity accounts for 20% of the base score. This weighting ensures that while the engine optimizes for semantic similarity and distinct categories, highly rated and highly trafficked attractions naturally bubble up, preventing the system from overly recommending overly obscure places when an iconic landmark is a perfect match!

**Q: If the place is closed, zero the final score.**
This is structured as an optimized **two-stage defense system**:
1. **Stage 1: SQL String Extraction**: Instead of loading thousands of entries into Python memory, the PostgreSQL query acts as a bouncer. It runs a Regex validation (`^\d{1,2}:\d{2}-\d{1,2}:\d{2}$`) to ensure the hours string is cleanly parseable. If it's valid, it uses `SPLIT_PART` to mathematically slice the string into exact Opening and Closing integers. If the user's `current_hour` falls outside that threshold (or inside an inverted overnight block like `22-02`), the candidate is outright deleted from the database return.
2. **Stage 2: Python Safety Override**: If a strange, messy Foursquare string (like `"12 PM: Open"`) bypasses the strict SQL Regex filter, it gets loaded into Python memory. The `RankingEngine` manually re-evaluates it. If an attraction is verifiably and unequivocally **CLOSED**, Python invokes a Hard Override parameter taking the candidate's entire mathematical `final_score` to absolute `0.0`, burying it at the bottom.

**Q: Travel style is a string. Keep it in mind for POC.**
Noted! `travel_style` expects strings like `"budget"`, `"balanced"`, or `"premium"`. We will explicitly program the budget score to align these user labels against the attraction's `budget` field (which might be symbols like `"$"` or integers), mapping string intents to hard symbols.

**Q: Exclude explicit visited categories to use for Diversity.**
Currently, the diversity bonus looks at what categories have appeared *so far in the current candidate list* as it iterates. To accurately reward purely novel experiences, we will query the `trip_feedback` table in `client_info`, cross-reference those `place_ids` with the `attractions` table to get their exact categories, and pass that `real_seen_categories` set directly into the `RankingEngine` when it initializes.

**Q: Please explain penalty and diversity score.**
This is the organic re-sorting mechanism!
- **Positive Diversity**: If an attraction has a category the user hasn't seen yet (e.g., "Museum" when all previous recommendations were "Cafes"), we add `+0.05` to their score.
- **Negative Cluster Penalty**: We track how many times we've recommended something from `location_cluster_id = 33`. The first attraction from Cluster 33 gets `0` penalty. The second attraction gets a small penalty (e.g., `-0.05`). The third gets `-0.10`. 
This mathematical penalty ensures that a slightly less relevant attraction in a *different* neighborhood (Cluster 40) will mathematically overtake the third attraction in Cluster 33, naturally forcing the engine to recommend a diverse geographic spread!

### Real-Time Feedback Loop

**Q: What is the difference between visited and skipped? Why don't we give it attention?**  
They both serve identically as purely *negative filters*. When a user clicks "skip" or "visit", the system logs that place into `trip_feedback` merely so it can be blocked (`NOT IN`) from ever appearing again on the trip. We intentionally DO NOT update the AI vector on a "skip" because shifting the vector mathematically *away* from a skipped place is extremely risky. For example, if a user skips a highly-rated pizza place solely because they aren't hungry right now, mathematically moving their entire user personality *away* from Italian Food ruins the integrity of their profile. We only reward positive signals; we don't attempt to punish negative ones mathematically.

**Q: What does `ON CONFLICT (trip_id, place_id) DO UPDATE` do? (`feedback_queries.py:L142`)**  
This is PostgreSQL's exact native syntax for an "Upsert". The database table `trip_feedback` has a `UNIQUE` combination lock on `(trip_id, place_id)`—you physically cannot have two records for the exact same attraction on the same trip. If a user "skips" a place, but later changes their mind and "likes" it, the database hits a conflict trying to insert the second row. Instead of crashing Python, `DO UPDATE SET action = EXCLUDED.action` gracefully overwrites the action from "skipped" to "liked".

**Q: 0.3 for EMA_ALPHA isn't extremely high? If I go to Cafe once my vector changes entirely. (`preference_service.py:L180`)**  
It may seem inherently high, but Exponential Moving Averages (EMA) work linearly against the *remaining* weight (`1 - EMA_ALPHA`), which is 0.7. So the new Cafe embedding is worth exactly 30%, while the user's core massive foundational profile still owns a powerful 70% of the mathematical gravity. 30% is high enough to confidently skew the upcoming cluster searches to rapidly notice Cafes nearby, but mathematically not high enough to completely erase their original trip setup (like Museums or Parks) out of the vector space. It ensures the app is highly responsive to real-time mood swings without instantly inducing amnesia!
